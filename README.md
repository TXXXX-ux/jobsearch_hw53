## 🛠 Реализация регистрации и обработки ошибок

В рамках этой части проекта была полностью переписана логика регистрации пользователей. Теперь система не просто записывает данные, но и следит за их уникальностью.

### Что было сделано:

* **Repository Layer**: В `UserRepository` добавлена поддержка кастомных запросов. Теперь мы умеем проверять существование email в базе (`existsByEmail`) и искать юзера для авторизации (`findByEmail`).
* **Service Layer**: В `UserService` внедрена бизнес-логика. Перед сохранением нового пользователя сервис проверяет, не занята ли почта. Если занята — выбрасывается `RuntimeException`, предотвращая дубликаты.
* **Controller Layer**: Обновлен `UserController`. Все эндпоинты (`/users`, `/users/search`, `/users/register`) теперь связаны с сервисом и возвращают актуальные данные.
* **Global Exception Handling**: Чтобы сервер не "плевался" красными логами в консоль, добавлен глобальный обработчик ошибок. Теперь при попытке повторной регистрации клиент получает чистый ответ `400 Bad Request` с текстом "Email уже занят!".

Как это выглядит в деле:
Действие	Результат	Статус
Первая регистрация	Пользователь сохранен в H2	200 OK
Повторная регистрация	Сообщение "Email уже занят!"	400 Bad Request
Поиск по Email	Возврат JSON с данными юзера	200 OK
